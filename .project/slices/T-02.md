# Slice T-02 — Config-driven coverage snapshot + server ingest

## Why
The dashboard coverage view shows only a ratio + three counts. But the config
already DECLARES which files each doc describes, so we can show the **full list**:
every file, marked documented / undocumented / waived, with its owning doc(s).
`coverage.resolve_coverage(config, inventory)` already computes exactly this. This
slice (1) defines the serializable **wire snapshot** carrying that full file list,
and (2) gives the central server a route to INGEST a coverage snapshot so the
dashboard can render it. (T-01's `ReviewRecord.ticket` already rides the existing
`GET /records` response — no server change needed for tickets.)

## In scope
### `code_doc_monitor/coverage.py` — `coverage_snapshot(report) -> dict` (pure, K10)
Add a function (NOT a CLI-only helper — it is a documented public symbol):
```
def coverage_snapshot(report: CoverageReport) -> dict:
```
Returns a deterministic, JSON-safe dict that is the **wire shape** the server
stores and the dashboard reads:
- `schema_version: "1.0.0"`
- `percent_files: float`, `percent_public_symbols: float`
- `ratio: float` = `percent_public_symbols / 100` — **REQUIRED for back-compat**:
  `server/app.py::_compute_status` reads `snapshot["ratio"]` for `RepoStatus`.
- `documented: int`, `undocumented: int`, `waived: int` — FILE basket counts
  (`len(report.documented_files)` etc.) so the existing dashboard baskets keep working.
- `files: list[dict]`, one per file in `report.files` (already sorted by path), each:
  `{"path": f.path, "language": f.language, "owners": list(f.owners),
    "status": "documented" if f.owners else ("waived" if f.waived_reason else "undocumented"),
    "waived_reason": f.waived_reason}`.
Keep it dependency-free and deterministic (files already sorted by path, K10).
Add `coverage_snapshot` to `__all__`.

### `code_doc_monitor/server/app.py` — `POST /repos/{repo_id:path}/coverage`
Add a WRITE route mirroring `resolve` (token-protected, K8):
- Path: `@app.post("/repos/{repo_id:path}/coverage", status_code=202)`.
- Body: accept the snapshot. Define a tiny typed model in app.py for validation —
  `class CoverageIngest(BaseModel)` with `model_config = ConfigDict(extra="allow")`
  and a required `ratio: float | None = None` is too loose; instead accept the
  snapshot as a permissive model: a `captured_at: str` field (injected ISO from the
  client, K10) plus `model_config = ConfigDict(extra="allow")` to carry the rest
  (`files`, baskets, percents). Store the FULL dumped dict.
  - Implementation: `_require_known_repo`; `_verify_token`; then
    `store.add_coverage_snapshot(repo_id, payload.captured_at, payload.model_dump(mode="json"))`.
    Return `{"repo_id": repo_id}`.
- Add `CoverageIngest` to `app.py::__all__`.
- The existing `GET /repos/{id}/coverage` already returns `store.coverage_for(repo_id)`
  (the list of stored snapshot dicts) — unchanged; it now carries `files`.

### Store
`InMemoryStore.add_coverage_snapshot(repo_id, captured_at, snapshot)` already
exists and is enough. The `Store` Protocol does NOT currently declare
`add_coverage_snapshot` — add it to the `Store` Protocol (so the route can call it
through the seam) and confirm `db.py`'s SqlStore implements it (check `tests/test_db.py`;
if SqlStore lacks it, add a minimal parity impl OR keep the route using the concrete
method via a `getattr` guard — PREFER adding it to the Protocol + SqlStore for K6
cleanliness). If SqlStore changes, keep it tested.
