# Slice Z-01a — nested `dir-covered` (deepest-wins attribution)

Prerequisite for Z-01b (dogfooding cdmon's own flat package into units). Relaxes
the dir-covered disjointness rule to allow NESTING and attributes each file to the
deepest matching unit. See `.project/spec/CONFIGV2.md` §1.1.

## Why

cdmon's engine is a flat `code_doc_monitor/` package with 12 documents, plus the
`agent/` and `server/` subpackages. The natural unit split is: a `core` unit
covering `code_doc_monitor` and child units covering `code_doc_monitor/agent` and
`code_doc_monitor/server`. Today's "dir-covered must be disjoint (loud on
overlap)" rule forbids a parent+child pair. The general, correct model (and the
one the user's `agent/` example implies) is: **nesting is allowed; a file is
attributed to the DEEPEST unit whose `dir-covered` contains it.** Only two units
with the IDENTICAL dir-covered path genuinely conflict.

## Goal (validable)

`load_bundle` accepts units whose `dir-covered` nest (parent + child) and rejects
ONLY identical/duplicate dir-covered paths. Coverage attribution, `.rpt`
per-unit breakdown, and `suggested_unit` all use deepest-wins: a file under both a
parent and a child unit's dir-covered belongs to the CHILD. The per-extension
`source-files-format` scoping uses the deepest unit's formats. Proven by tests:
nested units load; a file in the child dir attributes to the child unit (not the
parent); a file directly in the parent dir attributes to the parent; identical
dir-covered across units still loud; existing disjoint-sibling behavior unchanged.

## In scope

- `code_doc_monitor/config.py`:
  - Relax the overlap check in `load_bundle`: nesting (one dir-covered is an
    ancestor of another) is ALLOWED. Loud `ConfigError` ONLY when two units share
    an IDENTICAL dir-covered path (normalized). Keep determinism + clear message.
  - Add `unit_for_path(bundle, repo_relative_path) -> UnitFile | None`: the unit
    whose dir-covered is the DEEPEST ancestor of the path (longest matching
    directory prefix by path components). Used by attribution. (Add to
    ConfigBundle as a method too if convenient.)
  - `effective_coverage`: the include set is still the union of dir×format globs
    (a parent glob already covers children — fine). The EXCLUDE/scoping for
    format must reflect the DEEPEST unit: a file under a child unit whose
    extension is not in the CHILD's source-files-format is excluded even if the
    parent would have included it (and vice-versa). Implement this via a derived
    per-file decision using `unit_for_path` — DO NOT change coverage.py/inventory.
    If the current include/exclude glob approach can't express deepest-wins format
    scoping precisely, add a post-filter step in effective_coverage's contract
    (document it) OR compute the include globs from per-deepest-unit formats so the
    universe is exactly right. Add a test proving a child format mismatch is
    excluded while the parent format would have kept it.
- `code_doc_monitor/report.py`: per-unit attribution + `suggested_unit` use
  `unit_for_path` (deepest-wins). A file in a child dir is counted under the child
  unit only; the parent unit does not double-count it. Update/extend tests.
- Update `.project/spec/CONFIGV2.md` §1.1 ("disjoint" → "nesting allowed,
  deepest-wins; identical paths conflict") + ARCHITECTURE.md.

## Test plan
- config: nested units load OK; identical dir-covered loud; deepest-wins
  `unit_for_path` (child file→child, parent file→parent, unrelated→None);
  sibling-disjoint unchanged.
- effective_coverage: child format-mismatch excluded though parent format would
  include; universe correct with nested units.
- report: per-unit counts don't double-count child files; suggested_unit picks the
  deepest unit.
- Full suite green (existing N-01/N-03/N-04 tests must still pass — the disjoint
  cases are a subset of "no identical paths"). ≥90% branch coverage of changed code.
