# Slice T-04 — End-to-end / system / integration + live seed

## Why
Prove the whole vertical works through every layer, then make the LIVE instance on
:33333 actually SHOW the new ticket + config-driven coverage so the user can see it.

## In scope
### End-to-end / system test (Python) — `tests/test_system.py` (extend) or new `tests/test_e2e_ticket_coverage.py`
A single test that exercises the FULL pipeline OFFLINE (mock backend) and through
the FastAPI app via `fastapi.testclient.TestClient`:
1. Build a tiny fixture repo (reuse the `_SHARED_V1`/doc-stub pattern already in
   `test_system.py`) with one eng-guide doc over one code file. Run `Monitor.run`
   with a fixed `now` ⇒ get `records` each carrying a non-None `ticket`.
2. Build a `CoverageReport` via `resolve_coverage(config, inventory)` over the
   fixture and `coverage_snapshot(report)` ⇒ a snapshot dict with `files`.
3. `create_app(InMemoryStore())` → TestClient. `POST /repos` (register with a token),
   `POST /ingest` each ticketed record (Bearer), `POST /repos/{id}/coverage` the
   snapshot (Bearer).
4. ASSERT the round trip:
   - `GET /repos/{id}/records` → each record's `ticket` survived with `ticket_id`,
     `severity`, `acceptance_criteria`, `affected_symbols` intact.
   - `GET /repos/{id}/coverage` → last snapshot carries `files` with documented vs
     undocumented status matching the config (the doc'd file is "documented", an
     unreferenced file is "undocumented").
   - `GET /repos/{id}/status` → `coverage_ratio` equals the snapshot ratio.
   - Auth negative path: `POST /ingest` / `POST /coverage` without the token ⇒ 401.
This is the integration + system + e2e proof in one deterministic test (K10).

### Full gate (the "ralph loop" — iterate until ALL green)
- Python: `.venv/bin/ruff format --check`, `.venv/bin/ruff check`,
  `.venv/bin/mypy custodex`, **reheal**
  (`.venv/bin/cdx monitor --apply --config cdmon.yaml`), then
  `.venv/bin/pytest` (≥90% fail_under) — all green.
- Dogfood: `.venv/bin/cdx check --config cdmon.yaml` and `lint` exit 0;
  `.venv/bin/cdx coverage --config cdmon.yaml --fail-under 95` passes.
- Frontend (from `dashboard/`): `npm run lint`, `npm run build`, `npm run test:run` green.

### Live seed + restart (so the user SEES it)
- Write a small, committed seed helper `scripts/seed_demo.py` (create `scripts/` if
  absent; it is NOT under `custodex/**` so it is out of the coverage scan)
  that, given a running base URL (default `http://127.0.0.1:33333`) OR a store,
  registers a couple of demo repos, ingests a few **ticketed** ReviewRecords (run
  `Monitor` over this repo, or synthesize realistic records with full tickets), and
  posts a real `coverage_snapshot` built from THIS repo's own
  `resolve_coverage(load(cdmon.yaml))` — so the dogfood file list shows up.
  - Keep it deterministic (inject a fixed `now`/timestamps).
  - The server uses an in-memory store, so seeding posts over HTTP to the live
    process (use stdlib `urllib`, no new dep) OR builds a pre-seeded store and
    relaunches. PREFER: a `create_seeded_app()` path the launcher can use, so a
    restart comes up already populated (cleaner than racing HTTP against a live one).
- Rebuild the dashboard (`npm run build`) so `/assets` has the T-03 bundle, then
  restart the unified server on 0.0.0.0:33333 serving the seeded store, and verify
  with curl: `/repos`, a repo's `/records` (ticket present), `/coverage` (files
  present), `/status`, `/`, `/assets/*`, `/docs` all 200.

## Test plan
The e2e test above IS the system/integration test. Also assert `scripts/seed_demo.py`
imports cleanly (a smoke test) if it carries reusable functions (e.g. a
`build_demo_records()` that the e2e test can share).

## Gate
Everything above green. Report the live URL + a one-line proof for each of:
ticket round-trip, coverage file list, all endpoints 200.

## Out of scope
New product features. Postgres persistence (still InMemoryStore — note it as the
known caveat). Per-page visual polish beyond T-03.
