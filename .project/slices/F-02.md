# Slice F-02/F-03 — drift timeline + coverage views

EPIC F. Two read views over the EPIC-E API, reusing F-01's `ApiClient` + types.
Adds routing (a repo detail route). Frontend-only; Vitest gate.

## Goal (validable)
(F-02) A per-repo **detail** view lists that repo's records as a drift/action
timeline (doc, drift_kind, verdict, detected_at, source_sha) with filter controls
(verdict / drift_kind / audience), and shows each record's resolution if present;
clicking a repo on the Repos page routes to it. (F-03) A **coverage** view renders
the latest coverage snapshot (% documented + the documented/undocumented/waived
baskets). Both fetch from the API (mocked in tests) with loading/error/empty
states. Proven by Vitest tests asserting rendered rows + filter behavior + the
coverage numbers, plus a clean build.

## In scope (`dashboard/`)
- Add `react-router-dom`; routes: `/` (Repos, F-01), `/repos/:repoId` (detail,
  F-02), `/repos/:repoId/coverage` (F-03). Update `App.tsx` with the router.
- `ApiClient` gains: `recordsFor(repoId, filters?: {verdict?, drift_kind?,
  audience?, doc_id?, limit?, offset?}) → ReviewRecord[]`, `resolutionsFor(repoId)
  → ResolutionRecord[]`, `coverageFor(repoId) → CoverageSnapshot[]` (the GET
  endpoints from E-05; reads are OPEN). Filters become query params.
- TS types for `ReviewRecord`/`ResolutionRecord`: GENERATE from `cdx schema`
  (run `.venv/bin/cdx schema --out dashboard/src/schema.review.json` then a tiny
  json-schema→TS step OR hand-write the fields you render and note it). Keep it
  honest: the rendered fields must match the real schema.
- `src/pages/RepoDetail.tsx`: timeline list + filter controls (controlled inputs →
  re-query). Join resolutions by `record_id` to show outcome badges.
- `src/pages/Coverage.tsx`: % + three baskets from the snapshot.
- A shared `useApi`/loading-error hook (small) to avoid repeating fetch boilerplate.

## Test plan (Vitest)
- RepoDetail renders records from a mocked client (assert doc + verdict + a
  resolution badge for a resolved one); changing a filter triggers a re-query with
  the right params (fake client captures the call args); loading/error/empty.
- Coverage renders % + basket counts from a mocked snapshot; empty snapshot → a
  friendly "no coverage reported yet".
- routing: clicking a repo navigates to `/repos/:id` (Testing Library + memory
  router).
- `npm run lint`, `npm run test:run`, `npm run build` all green.
