# Slice R-03 — Traceability engine (feature ⇄ demo/test/source) + `cdmon trace`

EPIC R, phase R2. The catalog (R-01/R-02) names every feature; R-03 builds the
engine that PROVES each feature is exercised — at least one demo AND at least one
test — by scanning for an inline feature-tag convention and producing a coverage
matrix + gap report. This is the backbone R3 (demo mapping) and R5 (test mapping)
fill in, and the gate R-07 wires into CI.

## Goal (validable)
1. **Tag scan.** `scan_refs(root, kind)` finds every `Feature: <id>[, <id>...]`
   (and `Features:`) tag in `.py`/`.md` files under `root`, returning sorted
   `FeatureRef(feature_id, path, kind, line)`. A bare `FEAT-id` mention WITHOUT the
   `Feature:` marker is NOT a reference (the marker disambiguates evidence from
   prose). Pure, never imports the scanned file (K1).
2. **Matrix.** `build_matrix(catalog, tests_root=, demo_root=, source_root=None)`
   returns a `TraceMatrix` with `tests_for`/`demos_for`, `features_without_test`,
   `features_without_demo`, `unknown_refs` (a tagged id not in the catalog → a loud
   gap, K8), and `is_complete`.
3. **Gap fixture.** On a fixture where F1 is tagged in a test AND a demo and F2 is
   tagged nowhere: `tests_for(F1)`/`demos_for(F1)` non-empty; F2 in BOTH
   `features_without_test` and `features_without_demo`; a test tagging a bogus
   `FEAT-NOPE-999` shows in `unknown_refs`. Deterministic ordering (K10).
4. **CLI.** `cdmon trace` loads `feature-doc/catalog`, scans `tests/` + `demo/`,
   prints a summary; `--json` emits the matrix; `--fail-on-gap` exits nonzero iff
   not `is_complete`. System test over a tmp fixture tree.
5. **Render.** `render_matrix_md(matrix)` is a pure, byte-stable traceability table
   + gaps section.

## Design
See ARCHITECTURE.md `traceability.py` pin. Frozen pydantic models (`extra=forbid`),
module-level compiled regex (`FEATURE_REF_RE`, `_TAG_RE`), sorted everything (K10).
`EvidenceKind` enum TEST/DEMO/SOURCE. No new dependency (K0). `cli.py` adds the
`trace` command mirroring the existing command patterns (`_load`, exit codes).

## Schema / migration (K6)
None — new self-contained module + a new CLI command. No wire/schema change.

## Test plan (`tests/test_traceability.py`, new — TDD red first)
- **unit:** `_TAG_RE` matches `Feature: FEAT-X-001`, `Features: FEAT-X-001, FEAT-Y-002`,
  case-insensitive marker; does NOT match a bare `FEAT-X-001` in prose; multiple ids
  on one line split correctly. `scan_refs` on a tmp tree returns sorted refs with
  correct path/line/kind; ignores non-.py/.md; recurses. `TraceMatrix` accessors +
  `is_complete` logic; `unknown_refs` catches a non-catalog id.
- **integration:** `build_matrix` over a fixture (catalog + tagged test + tagged
  demo) — F1 covered, F2 gap both ways, bogus id in unknown_refs. `render_matrix_md`
  pure/stable, contains each id + a Gaps section.
- **system/e2e (test_system.py or in-file):** invoke `cdmon trace --fail-on-gap`
  over a tmp fixture via the Typer CLI runner — exit 0 when complete, nonzero with a
  gap; `--json` parses and reflects the matrix.
- **coverage:** every accessor + both CLI exit branches for ≥90% branch.

## Dogfood
`traceability.py` is a new public module → add it to the coverage waiver (reason:
EPIC R, api doc owned by R-07 wiki), exactly like `featurecatalog.py` in R-01.
Adding `cdmon trace` to `cli.py` drifts the cli api doc → reheal with
`cdmon monitor --apply`; confirm `cdmon check`/`lint`/`coverage --fail-under 95` exit 0.
NOTE: do NOT wire `cdmon trace --fail-on-gap` into the gate yet — tests/demos are
not annotated until R-04/R-05, so it would (correctly) report ~100% gaps. R-03 only
ships the engine + command; the CI gate is R-07.

## Out of scope
Annotating real tests/demos (R-04/R-05); test-function-level / demo-case-level
attribution (R-05 AST parse refines file-level → node-level); source inline tags
(R-06); the `cdmon wiki` regeneration + CI gate (R-07).

## Constraints
K0 (no new dep), K1 (scan reads files as text, never imports them), K8 (unknown
ref is a loud reported gap; `--fail-on-gap` nonzero exit), K10 (sorted, deterministic
scan + render).
