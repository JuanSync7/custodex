# Slice R-06 — Test annotation + test wiki (`testwiki.py`)

EPIC R, phase R5. Completes the TEST side of traceability and gives a navigable
test wiki generated from the tests' own docstrings (no second source of truth).
Two parts: (a) the `testwiki.py` extractor engine (TDD); (b) an annotation pass
that tags every test file with the catalog features it covers, so every feature
has ≥1 test.

## Goal (validable)
1. **Extractor.** `collect_tests(tests_root)` AST-parses the test tree (NEVER
   imports it — K1), returning `TestModule`/`TestCase` records with nodeid, path,
   name, boundary (from the R-05 directory), docstring summary, and `Feature:`
   refs (per-test + inherited module-level). Deterministic, sorted (K10). See the
   ARCHITECTURE.md `testwiki.py` pin for exact signatures.
2. **Render.** `render_test_wiki_md(modules)` is a pure, byte-stable wiki grouped
   by boundary → module with per-case summaries + feature links, plus a per-feature
   "tested by" index.
3. **Test coverage complete.** After annotation,
   `build_matrix(catalog, tests_root=tests, demo_root=demo).features_without_test()`
   is EMPTY and `unknown_refs()` is EMPTY — every feature has ≥1 test, no test tags
   a non-existent feature. (This + R-04's demo side makes `cdmon trace --fail-on-gap`
   green — wired into the gate in R-08, not here.)

## Design
- **Part A — `code_doc_monitor/testwiki.py`** (new): `TestBoundary` enum,
  `TestCase`/`TestModule` frozen pydantic, `collect_tests`, `render_test_wiki_md`.
  Reuse `traceability`'s `_TAG_RE`/`FEATURE_REF_RE` for feature parsing (import from
  traceability — no duplication). Boundary from path (`tests/unit/…`→UNIT, etc.).
  Pure stdlib `ast`; robust (a test with no docstring → empty summary), loud only on
  an unparseable file (K8).
- **Part B — annotation.** Add a module-docstring `Features: <ids>` line to each
  `tests/{unit,integration,system}/test_*.py` (and smoke/regression where they map)
  listing the catalog features that file exercises. File-level granularity: honest
  (the file tests those features) and tractable. Done as a fan-out, one subagent per
  boundary/group, each reading the test file + the relevant catalog subsystem to map
  honestly. The traceability TEST side then completes.

## Test plan
- **unit (tests/unit/test_testwiki.py):** `collect_tests` over a tmp test tree —
  finds top-level + class-nested `test_*`, pulls docstring summary, resolves boundary
  from path, gathers per-test + module-level `Feature:` refs; a no-docstring test →
  empty summary; an unparseable file → loud (K8); deterministic order.
  `render_test_wiki_md` pure/stable, contains boundary headings + a feature index.
- **integration:** `collect_tests(Path('tests'))` over the REAL tree returns the
  expected module count and every case carries a valid boundary (not UNKNOWN).
- **system (tests/system/test_test_coverage.py or similar):** assert
  `build_matrix(...).features_without_test() == ()` and `unknown_refs() == ()` over
  the real annotated tree — the "every feature has a test" guarantee, automated.
- Full gate green; new module added to the dogfood coverage waiver (R-07 wiki owns
  its api doc); `cdmon check`/`lint`/`coverage --fail-under 95` exit 0.

## Out of scope
The `cdmon wiki` regeneration command + wiring `cdmon trace --fail-on-gap` into CI
(R-08); the source index + source wiki (R-07); rewriting every test's prose (we read
existing docstrings — only ADD `Features:` tags + enrich where a docstring is absent
and a test is feature-critical).

## Constraints
K0 (no new dep — stdlib ast + existing traceability regex), K1 (AST parse, NEVER
import/exec a test), K8 (loud on an unparseable test file; unknown feature ref
caught by traceability), K10 (deterministic sorted collect + render). Tags are
honest: a file is tagged with a feature only if its tests exercise it.
