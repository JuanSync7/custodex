# Slice D-05/D-06 — promotion detector + deterministic rule application

Final EPIC-D slices. The self-improvement payoff: when humans resolve the SAME
drift shape the SAME way ≥K times, promote it to a deterministic rule so the LLM
is no longer consulted for it — the system's cost curve bends DOWN as it learns.
Builds on D-01/02 (resolutions) and reuses similar.py's feature notion.

## Goal (validable)
(D-05) `detect_promotions(records, resolutions, min_count=3)` returns a
`PromotionCandidate` for each GENERALIZABLE shape `(doc_id, drift_kind, audience)`
whose resolved records ≥K ALL share one resolution (e.g. all `invalidated`);
deterministic. (D-06) with that promoted rule active, a matching drift is resolved
by the rule with **ZERO backend calls** (proven by a spy backend asserting it was
never called), while a non-matching drift still goes to the backend; rules are
opt-in (default none ⇒ byte-identical to today). Proven by a detection test
(3 identical → 1 candidate, 2 → none, mixed-resolution → none) + a monitor test
with a spy backend (matched shape → 0 calls, INVALIDATE verdict recorded).

## In scope
### D-05 — `code_doc_monitor/promotion.py` (new, pure)
```python
class PromotionCandidate(BaseModel):    # frozen, extra="forbid"
    doc_id: str
    drift_kind: str
    audience: Audience
    resolution: Resolution              # the unanimous human decision
    count: int                          # how many resolved records support it

def detect_promotions(
    records: list[ReviewRecord],
    resolutions: list[ResolutionRecord],
    *, min_count: int = 3,
) -> list[PromotionCandidate]: ...
```
- Shape key = `(doc_id, drift_kind, audience)` — GENERALIZABLE (NOT `surface_hash`,
  which is exact-state and never recurs; document this). Population = RESOLVED
  records (`resolved_index`). A shape qualifies iff ≥`min_count` of its resolved
  records share ONE resolution (unanimous among that shape's resolved records).
  Deterministic sorted output (K10).
- **Which resolutions are promotable:** `invalidated` and `rejected` are pure
  DECISIONS (no content) → safe to automate. `accepted` of a *mechanical* fix is
  already LLM-free. `overridden` carries human prose (`resolved_text`) that rarely
  generalizes — EXCLUDE it from auto-promotion (note it; a future content-rule
  slice could handle it). So promote only decision-shaped resolutions; document.

### D-06 — rule application in `monitor.py` (opt-in, additive)
```python
class PromotionRule(BaseModel):         # frozen, extra="forbid"
    doc_id: str; drift_kind: str; audience: Audience; verdict: Verdict
def rule_for(drift: Drift, rules: tuple[PromotionRule, ...]) -> PromotionRule | None
```
- `Monitor(..., rules: tuple[PromotionRule, ...] = ())` — default empty ⇒ today's
  behavior byte-identical. When a drift matches a rule, synthesize a
  `BackendResult` (the rule's verdict, e.g. INVALIDATE with `fix=None`, a `cause`
  like "promoted rule: <shape> resolved <verdict> by humans ≥K times") **WITHOUT
  calling `backend.propose`**, record it (mark it rule-sourced — e.g. `cause`
  prefix or a `backend_kind`-ish marker in the record/`config_snapshot`), and
  carry on. Non-matching drifts → backend as today.
- A `PromotionCandidate` → `PromotionRule` is a trivial mapping (a helper).

### CLI (light)
- `cdmon promotions [--config] [--min-count N] [--json]` — print candidates from
  the local logs (read-only). (Loading rules into `monitor` from a file can be a
  later slice; D-06's validable goal is satisfied by the in-memory `rules=` path
  + the `promotions` view.)

## Test plan
- `detect_promotions`: 3 resolved records of one shape all `invalidated` → 1
  candidate (count=3); only 2 → none; 3 but mixed resolutions → none; `overridden`
  excluded; deterministic order; different `surface_hash` in the same shape still
  groups (proves generalization).
- monitor + SPY backend (records call count): a drift matching a `rules=` entry →
  spy NEVER called, record has the rule verdict + rule-sourced marker; a
  non-matching drift → spy called as usual. Default `rules=()` → spy called for
  everything (existing behavior).
- CLI `promotions`: prints candidates / `--json` parses; `--min-count` honored.
- Additive: ALL prior tests green. ≥90% new code; gate ≥90%.
