# Slice R-04 — Demo 1:1 feature mapping (`demo/DEMOS.md`)

EPIC R, phase R3. R-03 built the traceability engine; R-04 makes the DEMO side
of the matrix complete: every one of the 186 catalog features is demonstrated by
at least one demo case, proven by `cdmon trace` reporting `features_without_demo
== []` with zero unknown refs.

## Goal (validable)
1. **Demo catalog.** `demo/DEMOS.md` enumerates demo CASES (`DEMO-001`, …). Each
   case: a title, what it demonstrates, how to OBSERVE it (a `walkthrough.py`
   step, a `cdmon` invocation, a config/doc artifact, or the `seed_demo`/server
   path), and a `Features: <id>[, <id>...]` tag line listing ONLY the features it
   genuinely exercises.
2. **Full demo coverage.** `build_matrix(catalog, tests_root=tests, demo_root=demo)
   .features_without_demo()` is EMPTY, and `.unknown_refs()` is EMPTY (every demo
   tag is a real catalog id). Verified via a python probe + `cdmon trace`.
3. **Honest mapping.** A demo case tags a feature only if the scenario actually
   demonstrates it. Cases are organized by scenario, reviewable, not a single
   catch-all "everything" case. Each subsystem has ≥1 case anchored to a real,
   observable scenario.
4. **Runnable backbone unchanged.** `demo/walkthrough.py` still runs offline
   (test_demo_* stay green); DEMOS.md may reference its steps by id. If a small,
   cheap scenario closes a real gap, the walkthrough may be extended (kept offline,
   deterministic, tests green).

## Design
- New `demo/DEMOS.md` (the demo catalog + traceability tags). Markdown; the
  `Features:` lines are what `traceability.scan_refs(demo, DEMO)` picks up.
- Group cases by user journey: (a) detect→heal loop, (b) apply-fix, (c)
  link→generate (EDITOR), (d) coverage + gap→issue/ticket, (e) doctor/adopt, (f)
  config-v2 layout + index + sync, (g) central server + register + dashboard
  (`cdmon serve`/seed_demo), (h) backends/agent (offline MockBackend + agent
  graph), (i) record/log/sinks, (j) extractor seam + shell, (k) determinism /
  authority-mode / fingerprint-tier property cases. Tag each feature to the case(s)
  that show it.
- For features only observable via an opt-in path (live LLM, Postgres, server
  extra), the demo case DESCRIBES the exact observable steps (the offline
  walkthrough need not execute them) — a documented, reproducible demo case still
  counts as a demo. No dishonest tagging.

## Test plan
- **system (tests/test_demo_*.py or a new tests/test_demo_traceability.py):** assert
  `demo/DEMOS.md` exists; `scan_refs(demo, DEMO)` returns refs; every demo tag id is
  in the catalog (no unknown); `build_matrix(...).features_without_demo() == []`.
- **regression:** the existing test_demo_e2e / test_demo_walkthrough stay green
  (walkthrough still runs offline & deterministic).
- Reuse `cdmon trace` for the human check.

## Dogfood
DEMOS.md lives under demo/, not tracked source — no api-doc drift. `cdmon check`/
`lint` stay green. No coverage-waiver change.

## Out of scope
TEST side of the matrix (R-05/R-06 — tests aren't tagged yet, so full
`cdmon trace --fail-on-gap` stays red until then); the `cdmon wiki` regen + CI gate
(R-07). Node-level demo attribution.

## Constraints
K4 (the runnable demo stays offline/deterministic — no network/live LLM in the
walkthrough), K8 (no unknown feature ids in tags — caught by trace), K10
(deterministic walkthrough), K0 (no new dep). Tags are honest evidence, not
coverage theater.
