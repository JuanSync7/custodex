# Slice T-03 — Dashboard: DriftTicket card + config-driven coverage list

## Why
Surface T-01/T-02 to the operator. Replace the "one-liner" with a proper **ticket
card** the human reads and validates, and turn the coverage page from three counts
into the **full file list** (documented / undocumented / waived). Keep the existing
test hooks intact — these pages already have passing vitest suites.

Design system is the existing "DRIFT CONSOLE" (dark instrument panel; Fraunces +
IBM Plex Mono/Sans; mint=sync, amber=review, coral=drift, cyan=interactive). Reuse
the existing CSS vars and component classes in `src/index.css`; add new classes in
the same idiom (don't introduce a new palette/font).

## In scope
### `dashboard/src/types.ts` (ADDITIVE)
- Add `Severity = "low" | "medium" | "high"`.
- Add `AcceptanceCheck { text: string; auto_satisfied: boolean }`.
- Add `DriftTicket` mirroring `custodex/ticket.py::DriftTicket` fields
  exactly (ticket_id, title, summary, severity, drift_kind, doc_id, doc_path,
  region_id?, audience, affected_symbols: string[], root_cause, proposed_change,
  change_kind, diff, acceptance_criteria: AcceptanceCheck[], verdict, recommended_action).
- Add `ticket?: DriftTicket | null` to `ReviewRecord`.
- Add `files?: CoverageFile[] | null` to `CoverageSnapshot`, with
  `CoverageFile { path: string; language: string; owners: string[];
   status: "documented" | "undocumented" | "waived"; waived_reason?: string | null }`.
- Keep the "// server: …" provenance comments updated.

### `dashboard/src/pages/RepoDetail.tsx`
- Each record row becomes EXPANDABLE: clicking the doc cell (or a dedicated
  "View ticket" button) toggles a detail panel BELOW the row (a `<tr>` with a
  spanning `<td>`), rendering the record's `ticket` as a card when present:
  - A header: severity chip (`chip chip--{severity}` mapping high→drift/coral,
    medium→review/amber, low→sync/mint), the `title`, and a status badge derived
    from the resolution via the same logic as `ticket_status` (PROPOSED →
    "awaiting review"; VALIDATED/CHANGES_REQUESTED/REJECTED from the resolution).
  - Sections (use `<dl>`/labelled blocks): Summary, Root cause, Proposed change
    (+ `change_kind`), Affected symbols (mono chips), and a `diff` block (mono,
    pre) when non-empty.
  - An **Acceptance checklist**: render `acceptance_criteria` as a list; each item
    shows its `text` and an indicator for `auto_satisfied` (a mint dot/✓ "agent
    asserts" vs an amber "needs confirmation"). This is what the human validates.
  - The existing per-row resolve control stays — relabel its submit to **"Validate"**
    and keep the resolution `<select>` (accepted/overridden/rejected/invalidated)
    + note + token. (Resolving a ticket = validating it.) PRESERVE all current
    aria/test hooks on the resolve control and the table.
  - When `ticket` is null (old records), the panel shows the legacy `cause`/`fix.rationale`
    fallback so nothing regresses.
- PRESERVE existing test hooks: h1 "Drift timeline:", the filters/auth forms, the
  table with its column headers, `role="status"`/`role="alert"` states, the
  per-row resolution badge vs ResolveControl, the optimistic post behavior. Add the
  expand toggle as an additive control (give it an accessible name like
  `aria-expanded` + "View ticket"/"Hide ticket").

### `dashboard/src/pages/Coverage.tsx`
- Keep the % headline + the three baskets.
- BELOW them, when `latest.files` is present and non-empty, render a **file table**:
  columns `File` (mono path), `Status` (a signal dot + label: documented→mint,
  undocumented→coral, waived→muted/amber), `Owners` (doc id chips, "—" when none),
  `Reason` (the `waived_reason` for waived rows, else blank).
  - Add a tiny summary line: `f"{documented} documented · {undocumented} gaps · {waived} waived"`.
  - When `files` is absent (older snapshot), fall back to today's baskets-only view
    (no regression).
- Keep the existing loading/error/empty states and their `role` hooks.
