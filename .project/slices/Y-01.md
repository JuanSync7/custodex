# Slice Y-01 — DB tables + migration + Store methods for config documents/code-refs/sync

Part of EPIC Y (DB + sync). First Phase-2 slice. Pure persistence layer; the sync
ENGINE + endpoints are Y-02. See `.project/spec/CONFIGV2.md` §4.

## Goal (validable)

The central store can persist, replace, and query a repo's config documents +
code-refs (the relationship data) and a history of sync runs, with byte-for-byte
parity between `InMemoryStore` and `SqlStore`. Proven by extending the existing
store-parity HTTP/`store` parity test so every new method is asserted identical
across both backends, plus an alembic migration round-trip test (upgrade creates
the tables, downgrade drops them).

## In scope

Models (define near existing server models — `registry.py`/`store.py` — frozen
pydantic, additive K6):

```python
class ConfigDocument(BaseModel):     # one document as synced from config
    repo_id: str
    doc_id: str
    path: str
    audience: str
    unit: str | None                  # the unit file it came from (dir layout)
    region_keys: tuple[str, ...] = ()
    sync_kind: str                    # "git" | "local"
    ref: str | None = None            # branch/commit synced from
    synced_at: str

class ConfigCodeRef(BaseModel):      # one code_ref under a document
    repo_id: str
    doc_id: str
    path: str
    symbols: tuple[str, ...] = ()
    unit: str | None
    sync_kind: str

class SyncRun(BaseModel):            # one sync invocation summary
    repo_id: str
    sync_kind: str                   # "git" | "local"
    ref: str | None
    branch: str | None
    head_commit: str | None
    main_commit: str | None
    commits_ahead: int = 0
    fully_synced: bool
    document_count: int
    code_ref_count: int
    drift: dict                       # opaque drift/coverage summary (json)
    started_at: str
    finished_at: str
```

DB tables (`server/db.py`) — mirror the existing hybrid pattern (indexed scalar
columns + a full-JSON column as source of truth):
- `config_documents`: id PK, repo_id(idx), doc_id(idx), sync_kind(idx),
  path, audience, unit, ref, synced_at, `document`(JSON full ConfigDocument).
- `config_code_refs`: id PK, repo_id(idx), doc_id(idx), sync_kind(idx),
  path, unit, `code_ref`(JSON full).
- `sync_runs`: id PK, repo_id(idx), sync_kind(idx), ref, branch, head_commit,
  main_commit, commits_ahead, fully_synced(bool), `run`(JSON full SyncRun).

Alembic migration `alembic/versions/0003_*.py` creating the three tables (mirror
0001/0002 style); `downgrade` drops them. Add a migration round-trip test if the
suite has a pattern for it (find how 0001/0002 are tested).

Store methods (add to the Store Protocol + InMemoryStore + SqlStore; parity-
tested):
```python

                   documents: list[ConfigDocument],
                   code_refs: list[ConfigCodeRef]) -> None: ...
    # atomically delete existing rows for (repo_id, sync_kind) then insert the new set
def config_documents_for(self, repo_id: str, sync_kind: str | None = None) -> list[ConfigDocument]: ...
def code_refs_for(self, repo_id: str, doc_id: str | None = None,
                  sync_kind: str | None = None) -> list[ConfigCodeRef]: ...
def add_sync_run(self, run: SyncRun) -> None: ...
def latest_sync_run(self, repo_id: str, sync_kind: str | None = None) -> SyncRun | None: ...
def sync_runs_for(self, repo_id: str, sync_kind: str | None = None) -> list[SyncRun]: ...
```
Insertion-order determinism (K10): order by surrogate id, no name/time sort.
`replace_config` is the idempotent upsert primitive (delete-then-insert for the
(repo_id, sync_kind) scope only — never touches the other sync_kind's rows).

## Test plan

- Extend `tests/test_server_store_parity.py` (or a new `tests/test_store_config.py`
  parametrized over `["memory","sql"]` the SAME way): for each method, assert
  InMemory and Sql return identical results. Cover: replace_config inserts +
  REPLACES (second call with fewer rows leaves no stragglers; other sync_kind
  untouched); config_documents_for filter by sync_kind; code_refs_for filter by
  doc_id/sync_kind; add_sync_run + latest_sync_run (most-recent by insertion) +
  sync_runs_for; empty results; JSON round-trip of tuple fields (region_keys,
  symbols) and the opaque drift dict.
- Migration test: 0003 upgrade creates the 3 tables; downgrade drops them (match
  the existing 0001/0002 test pattern; run under the SQLite test path).
- ≥90% branch coverage of new code; full suite green (incl. `pg`-marked twin if
  the parity suite has one — keep the `pg` marker usage consistent).

## Out of scope
The sync ENGINE (reading git/working-tree config, computing drift) + the HTTP
endpoints — that's Y-02. The CLI `cdmon sync` + local_path registration — Y-03.

## Constraints
K6 (additive schema; full-JSON source of truth), K10 (insertion-order
deterministic), K8 (loud). Postgres-first with SQLite stand-in (honor the `pg`
marker convention).
