# Slice N-05 — writing templates + `doc-style.yaml` + agent authoring wiring

Part of EPIC N (CONFIG-V2). Builds on N-01. See `.project/spec/CONFIGV2.md` §1.4
and §2. Last Phase-1 slice.

## Goal (validable)

Four independent template categories exist as markdown under
`templates/writing/{document-type,tone,writing-style,vocabulary}/` (multiple
templates each). A `config/cdmon/doc-style.yaml` maps each document id to one
template per category (with `defaults`). The agent, when AUTHORING a document's
prose (the `llm`/no-renderer authoring path), composes the four selected template
files into its prompt. Proven by: `load_doc_style` parses + validates the map
(loud if a named template file is missing); `style_for(doc_id)` resolves the 4
files (mapping overrides defaults); and an agent-prompt test asserts the four
template bodies appear in the composed authoring prompt for a doc.

## In scope

`templates/writing/` (new content) — concrete, useful guidance files, e.g.:
- `document-type/`: `api-reference.md`, `tutorial.md`, `how-to.md`, `explanation.md`
- `tone/`: `precise.md`, `friendly.md`, `formal.md`
- `writing-style/`: `reference-dense.md`, `narrative.md`, `concise.md`
- `vocabulary/`: `engine-domain.md`, `general.md`
Each is a short (10–30 line) markdown brief of real authoring guidance for that
axis. (These are reference assets; keep them genuinely useful, not lorem ipsum.)

`code_doc_monitor/config.py` (or a small `code_doc_monitor/docstyle.py` if you
prefer to keep config.py lean — pick one, justify):
- `DocStyleFrontmatter` (cdmon-config-version=="2.0.0", kind=="doc-style-map",
  updated), `DocStyleSelection` (document_type/tone/writing_style/vocabulary —
  aliases `document-type` etc.), `DocStyleMapping` (doc + a DocStyleSelection),
  `DocStyleMap` (frontmatter, defaults: DocStyleSelection, mappings: tuple).
- `load_doc_style(path, *, templates_root: Path) -> DocStyleMap`: parse
  frontmatter+body; VALIDATE every named template resolves to an existing
  `templates/writing/<category>/<name>.md` (loud K8 listing the missing ones).
- `DocStyleMap.style_for(doc_id) -> DocStyleSelection`: mapping for the doc, else
  `defaults`.
- `resolve_style_files(selection, templates_root) -> dict[str, Path]`: the 4
  category→path mapping.
- `read_style_guidance(selection, templates_root) -> str`: the 4 file bodies
  concatenated with clear category headers (deterministic order:
  document-type, tone, writing-style, vocabulary).
- Wire into ConfigBundle: `load_bundle` also loads `doc-style.yaml` when present
  (pointer from index `doc-style`; default `doc-style.yaml`) with
  `templates_root` resolved (repo_root / "templates/writing"), exposing
  `bundle.doc_style: DocStyleMap | None`. Absent file → `None` (back-compat, K6).

Agent wiring — `code_doc_monitor/agent/`:
- FIRST read `agent/prompts.py`, `agent/backend.py`, `agent/runtime.py`,
  `agent/graph.py`, `agent/state.py` to find where the AUTHORING prompt for an
  `llm`/no-renderer region's prose is assembled.
- Add a seam so, when a DocStyleMap is available for the run, the composed
  authoring prompt for a given `doc_id` includes `read_style_guidance(...)`. Keep
  it OPTIONAL and additive (no style map → behavior byte-identical to today, K6);
  the offline MockBackend path must stay deterministic (K4/K10). Thread the
  guidance through the existing state/runtime rather than reaching into globals.
