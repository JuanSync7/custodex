# Slice E-01 — reporting client: repo identity + offline queue + retry

First EPIC-E slice. CLIENT-SIDE only (no server deps). Makes `HttpSink` a
robust multi-repo reporter: every record is wrapped in a **repo-identified
envelope**, sent with **bounded retry**, and on failure **queued to a local
outbox** that flushes when the transport recovers. All offline-tested via the
injected client (K4) — zero network.

## Goal (validable)
With a FAILING injected client, `HttpSink.emit` does not raise — it appends the
repo-identified envelope to a local outbox JSONL; when the client RECOVERS, the
next `emit` flushes the queued envelopes (oldest-first) before sending the new
one; envelopes carry `repo_id` + commit; bounded retry attempts are made before
queueing. Proven by: emit-while-down → outbox grows; emit-after-recovery → outbox
drains in order + the new record sent; envelope shape asserted; retry count
asserted; all with a fake client, NO network.

## In scope
### `config.py` — `CentralConfig` gains repo identity (additive)
```python
class CentralConfig(BaseModel):
    sink: Literal["none","file","http"] = "none"
    path: str | None = None
    url: str | None = None
    auth_env: str | None = None
    repo_id: str | None = None        # NEW — required when sink=="http" (loud K8 in make_sink if missing)
    repo_name: str | None = None
    repo_url: str | None = None
    outbox: str | None = None         # NEW — offline queue path; default ".cdmon/outbox.jsonl"
    max_retries: int = 2              # NEW — attempts before queueing
```

### `schema.py` (or `sinks.py`) — the wire envelope (shared, versioned, K6)
```python
class RepoIdentity(BaseModel):        # frozen, extra="forbid"
    repo_id: str; repo_name: str | None = None; repo_url: str | None = None
    commit: str | None = None

class IngestEnvelope(BaseModel):      # frozen, extra="forbid"
    schema_version: str = "1.0.0"
    repo: RepoIdentity
    record: ReviewRecord
```
This is the client↔server wire format (E-03 ingests it) — ONE schema, no DTOs.

### `sinks.py` — `HttpSink` upgrade (injected client, offline)
- Constructor gains `repo: RepoIdentity`, `outbox: Path | None`, `max_retries`.
- `emit(record)`: build `IngestEnvelope`; FIRST try to drain the outbox (post each
  queued envelope oldest-first, stop on the first failure and re-queue the rest);
  then send the new envelope with up to `max_retries` attempts; on final failure,
  append the new envelope to the outbox (never raise — reporting must not break a
  heal run). The injected client's `post` raising = "network down".
- Outbox is JSONL of `IngestEnvelope` lines (append-only; drained lines removed by
  rewriting the remainder — document the simple read-all/rewrite approach).
- `make_sink`: build `RepoIdentity` from `CentralConfig` (commit from
  `cfg`/`$CI_COMMIT_SHA` fallback — document precedence); loud `SchemaError`/K8 if
  `sink=="http"` and `repo_id` missing. NullSink/FileSink unchanged.

## Test plan (`tests/test_sinks.py`, extend)
- envelope shape: emit with a recording fake client → asserts the posted bytes
  decode to an `IngestEnvelope` with the right `repo_id`/commit + the record.
- down→queue: failing client → `emit` doesn't raise, outbox has N envelopes.
- recover→flush: after the client recovers, next `emit` posts the queued ones
  oldest-first THEN the new one; outbox empty after.
- partial flush: client fails on the 2nd queued item → first removed, rest re-queued.
- retry: client fails `max_retries`−1 times then succeeds → no queue, attempts counted.
- make_sink: http without `repo_id` → loud error; commit env fallback.
- K4: the real `_UrllibClient` POST stays the only `# pragma: no cover` leaf
  (cover the lazy-build via a stubbed leaf, as today).
- Additive: all prior sink tests green. ≥90% new code; gate ≥90%.
