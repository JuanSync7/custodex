# Slice G-03/G-04 — adopter CI templates + external example repo (e2e)

Final EPIC-G slices. Ships drop-in CI for adopters and PROVES the whole system on
a repo that is NOT the dogfood — an external example adopts cdx and reports to a
(TestClient) central server end-to-end.

## Goal (validable)
(G-03) Ship adopter CI templates (GitLab CI + GitHub Actions) under
`templates/ci/` that run `cdx doctor` → `cdx check` → `cdx lint` (gate) and a
post-merge `cdx sync-pr`/`open-docs-pr` (heal→docs-PR) job; they parse as valid
YAML and reference only real `cdx` subcommands. (G-04) An `examples/external-repo/`
fixture (its own tiny source + docs + `cdmon.yaml`) adopts cdx: a test drives
`init/register/check/monitor` against it and posts records to an in-process
TestClient central server, asserting the repo + records land server-side (incl.
the E-06 bearer write path). Proven by: a YAML-validity + subcommand-reference test
for the templates, and an e2e test wiring the example repo's HttpSink to a
TestClient-backed transport.

## In scope
### G-03 — `templates/ci/`
- `templates/ci/gitlab-ci.adopter.yml` — jobs: `cdmon-gate` (doctor + check + lint,
  on MR/branch) and `cdmon-docs-pr` (post-default-branch: `should-sync` guard →
  `sync-pr`/`open-docs-pr`), with commented env (`CDMON_CENTRAL_TOKEN`, central
  url). Mirrors our own `.gitlab-ci.yml` patterns.
- `templates/ci/github-actions.adopter.yml` — the same gate + docs-PR flow as a
  GitHub Actions workflow.
- A short `templates/ci/README.md` telling adopters how to use them (+ that the
  token is a CI secret).
- Test: both YAML files `yaml.safe_load` cleanly; a test asserts they reference only
  real `cdx` subcommands (parse the script lines, check against the CLI's command
  list) — so a renamed/removed command breaks the template test (keeps them honest).

### G-04 — `examples/external-repo/`
- A SMALL self-contained example: e.g. `src/widget.py` (a couple public functions)
  + `docs/api.md` (a managed `symbols` region) + a `cdmon.yaml` mapping them. NOT
  the dogfood; a stand-in for "some other team's repo".
- `tests/test_example_external.py`: build the example under `tmp_path` (copy the
  fixture), then:
  - `cdx check` finds it in/out of sync; `monitor --apply` heals it;
  - register the repo + emit its records to an in-process **TestClient** central
    server (wire the `HttpSink`'s injected client to call `TestClient`), with a
    bearer token (E-06) — assert `GET /repos` + `GET /repos/{id}/records` show the
    repo and the healed records server-side.
- This is the capstone integration: client config → heal → report → server stores
  → query — all offline via TestClient (no socket).

## Test plan
- templates: YAML valid; only-real-subcommands; the docs-PR job references
  `should-sync`+`sync-pr`/`open-docs-pr`; the gate job references `doctor`+`check`+`lint`.
- example e2e: heal converges on the external fixture; records reach the TestClient
  server with the right repo_id; an unauthorized emit (wrong token) is rejected
  server-side (proves the bearer path end-to-end).
- Additive: all prior tests green. ≥90% any new Python; gate ≥90%.

## Dogfood
No tracked-module change expected (templates + example fixture + a test). Confirm
`cdx check --config cdmon.yaml` exits 0. Update SPEC (a "Adopting cdmon" section)
+ README (point to `templates/ci/` + `examples/external-repo/`).

## Out of scope
EPIC H (self-improvement). Publishing to a package index. Real GitHub/GitLab runs
(templates are shipped + validated, not executed here).

## Constraints
K4 (the e2e uses TestClient, no socket/network), K0 (example needs no new dep), K6
(records cross the wire as the shared schema/envelope), K8, K10, K9. **Mark EPIC G
COMPLETE** when both land.
