## Goal (validable)
1. **Backend.** `GET /wiki` (global, public, no-auth) returns
   `{"sections": [{"id","title","html"}...]}` â€” the four committed wikis rendered to
   HTML via `build.render_markdown` (no new dep, K0), in deterministic order
   (features, traceability, tests, source). A missing file â†’ that section omitted; a
   missing/absent `wiki_dir` â†’ `{"sections": []}` (graceful). `create_app` gains an
   injectable `wiki_dir` (default `_wiki_dir()` = repo-root `feature-doc/`); tests
   inject a tmp dir (no real-FS coupling). FEAT-SERVER-019 added to the catalog +
   tagged on the server test + a demo case; `cdx trace --fail-on-gap` stays green;
   `cdx wiki` re-run; server api doc rehealed.
2. **Frontend â€” "show Wiki first".** AppShell shows a `Wiki` nav item (Reference
   group) â†’ `/wiki`; `pageLabel("/wiki")="Wiki"`. Present before any click; no wiki
   data loaded yet.
3. **Frontend â€” "switch on click".** `/wiki` routes to a `React.lazy` Wiki page in
   `<Suspense>` â€” the bundle + `api.wiki()` fetch fire only on navigation. The page
   renders the full wiki frontend: a section rail (4 sections, active state, count
   badge) + a prose pane rendering the selected section's HTML, with loading/error/
   empty states. Switching sections is client-side (one fetch).
4. **No new dependency** (server reuses render_markdown; dashboard adds no npm pkg).

## Design
See ARCHITECTURE.md `GET /wiki` pin. Mirror `/config/templates` (server) and the
`Config` page (frontend) patterns exactly. `WIKI_SECTIONS` tuple is the single
source of the section set/order. React.lazy + Suspense for the lazy switch.

## Out of scope
In-wiki full-text search/highlighting (a follow-on); per-section lazy fetch (all
sections come in one payload â€” the lazy boundary is the route); HTML twins of the
wikis; a Confluence export.
