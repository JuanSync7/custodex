# Slice E-05/E-06 — JSON query API + per-repo bearer auth

Final EPIC-E slices. Exposes the stored data for the dashboard (EPIC F) and
secures ingest with per-repo bearer tokens. Builds on E-03 (app/Store) + E-04
(SqlStore). All TestClient, offline.

## Goal (validable)
(E-05) `GET` endpoints return per-repo drifts/records (with filters:
verdict/drift_kind/audience/doc_id), resolutions, coverage snapshots, and a status
summary (counts, last-seen, escalation/unresolved tallies) — backed by the E-04
indexed columns. (E-06) `POST /ingest` (and `/repos` register) require a valid
per-repo bearer token; a missing/wrong token → 401/403; a valid token → accepted;
each repo's token is set at registration and stored hashed. Proven by TestClient:
filtered-query tests, a status-summary test, and auth accept/reject tests. NO
network.

## In scope
### E-05 — query API (`server/app.py` + Store/SqlStore read methods)
- `GET /repos/{repo_id}/records?verdict=&drift_kind=&audience=&doc_id=&limit=&offset=`
  — filter via the indexed columns (SQL WHERE), return `list[ReviewRecord]`
  (re-validated from the JSON column). Deterministic order (detected_at/id).
- `GET /repos/{repo_id}/resolutions` — `list[ResolutionRecord]`.
- `GET /repos/{repo_id}/coverage` — latest coverage snapshot (+ maybe history).
- `GET /repos/{repo_id}/status` — a summary model: total records, by_verdict,
  unresolved count, escalations, last detected_at, coverage % (if present). Define
  a small `RepoStatus` pydantic response model (this one MAY be a response DTO —
  it's a computed view, not the stored schema; document the distinction from K6's
  "no DTOs for the SHARED schema").
- Add the matching `Store` Protocol read methods (filtered `records_for`, status
  aggregation) — implement on BOTH `InMemoryStore` and `SqlStore` (keep the
  Protocol the one seam; the in-memory impl keeps the unit tests fast/offline).

### E-06 — per-repo bearer auth
- Registration mints/accepts a token: `RegistrationPayload` (or the `/repos` call)
  carries a token the repo will use; store it **hashed** (sha256) on the repo row
  (add a column/migration — additive). Decide+document: token provided by client
  vs server-minted-and-returned. (Recommend: client provides at register; server
  stores the hash; `/repos` returns confirmation.)
- A FastAPI dependency `require_repo_token` on `POST /ingest` (and re-register):
  read `Authorization: Bearer <t>`, hash, compare to the repo's stored hash →
  401 (missing) / 403 (mismatch) / pass. Unknown repo → 404 as before.
- The client (`HttpSink`/`registry`) ALREADY sends `Authorization: Bearer` from
  `auth_env` — verify the header name/format matches; adjust if needed.
- Alembic migration for the token-hash column (up/down tested).
