# Slice F-04/F-05 — resolve action (full-stack) + health overview

Final EPIC-F slices. Adds the FIRST write path from the dashboard (resolve a
record) and a health/metrics overview. SPANS server (new endpoints, Python) +
frontend (form + chart). Both gates apply.

## Goal (validable)
(F-04) `POST /repos/{repo_id}/resolutions` accepts a `ResolutionRecord` (the SHARED
schema), token-protected (E-06 write auth), persists via the Store; the dashboard
RepoDetail view has a resolve form that POSTs it (with the bearer) and the new
resolution appears. (F-05) `GET /repos/{repo_id}/health` returns a metrics summary
(escalation rate, unresolved/override counts, MTTR-style detected→resolved deltas
where both timestamps exist); the dashboard renders it. Proven by: server
TestClient tests (auth + persist + the metrics math) AND Vitest tests (form POST
with token, health render) — all offline.

## In scope
### Server (Python) — `code_doc_monitor/server/`
- `POST /repos/{repo_id}/resolutions` — body = `ResolutionRecord`; `require_repo_token`
  (401/403/404 like ingest); validates `record_id` exists for that repo (404 if
  not — loud K8); `store.add_resolution(repo_id, rec)`. Returns 202 `{record_id}`.
- `GET /repos/{repo_id}/health` → a `RepoHealth` view model (computed, like
  `RepoStatus`): `total`, `escalations`, `escalation_rate`, `unresolved`,
  `overrides`, `resolved`, and `mttr_seconds` (mean of resolved_at−detected_at over
  records that have a resolution; null if none). Add a `Store` aggregation method
  or compute from existing reads; implement on InMemoryStore + SqlStore.
- TestClient tests: POST resolution (right token→202+persisted; wrong→403; missing
  →401; unknown record→404); health math on a seeded fixture (escalation_rate,
  mttr) exact.
- mypy/ruff clean; the FULL Python gate stays green; whole-pkg ≥90%.

### Frontend (`dashboard/`)
- `ApiClient` gains `resolve(repoId, rec, token)` (POST with `Authorization:
  Bearer`) + `healthFor(repoId)`. A small POST helper.
- RepoDetail: a resolve control per unresolved record (pick resolution +
  optional text/note + a token field, or a token from a settings input) → POST →
  on success the row shows the new resolution badge (re-query or optimistic).
- A Health panel/page rendering the `RepoHealth` numbers (a simple bar/sparkline or
  just clear stat cards — keep deps lean; a chart lib is optional, not required).
- Vitest: resolve form POSTs with the right body + bearer (fake client captures
  it); success updates the UI; health renders the metrics; error/empty states.

## Test plan
- Server: the auth matrix + persistence + the metrics arithmetic (seed records
  with known detected/resolved timestamps → assert mttr_seconds + escalation_rate).
- Frontend: form → fake client called with `(repoId, ResolutionRecord, token)`;
  health view renders seeded numbers; missing token → a friendly validation, not a
  crash.
- Both gates green; migration unaffected (no new column — resolutions table exists).
