# Slice M-03 — make `demo/` a FULL feature gallery (heal loop, real gap, user-guide, varied templates, doctor)

Follow-up to M-01/M-02. Today `demo/` is a happy-path adopter setup: it is
pre-healed (`cdx check` always exits 0), 100% covered (the `.rpt`
`undocumented` list is always empty), uses only 2 of 12 writing templates,
eng-guide only, and never shows cdmon's CORE value — the detect→verdict→heal
loop. This slice makes the demo demonstrate the full surface while staying
fully functional and deterministic. See `.project/spec/CONFIGV2.md` §6.

## Goal (validable)

From `demo/`, a reader can SEE every headline cdx capability:
1. **A real coverage gap** — one genuinely undocumented source file appears in the
   committed `coverage.rpt` `undocumented:` list with a `suggested_unit`, and the
   dashboard Coverage page shows < 100%.
2. **A user-guide document** with `audience: user-guide`, mapped to the OTHER
   writing templates (tutorial / friendly / narrative / general) so all 4
   categories are exercised, not just api-reference/precise/reference-dense/
   engine-domain.
3. **The detect→heal loop** — a runnable, deterministic `demo/walkthrough.py`
   (offline, mock backend) that induces drift on a COPY of the demo, shows
   `cdx check` reporting drift, heals it with `cdx monitor --apply`, shows the
   review log, then runs `cdx rpt` (showing the gap) and `cdx doctor`.
4. The CANONICAL demo (the checked-in `demo/`) stays valid: `cdx check` exits 0
   (the gap is a coverage gap, NOT drift), `cdx lint` clean, `cdx rpt`
   idempotent (byte-identical re-render), `cdx doctor` passes. The central +
   standalone apps still serve it. Full suite green.

Proven by: the walkthrough test asserts drift-then-heal; the rpt test asserts the
undocumented entry + new percentages; the existing demo e2e tests are updated to
the new document/ref set and STILL pass; `cdx check`/`lint`/`doctor` from
`demo/` exit 0; `pytest -q -m "not live_llm and not pg"` green.

### 3. The detect→heal walkthrough (the core loop)
- Add `demo/walkthrough.py`: a deterministic, offline, import-safe Python script
  that DOES NOT mutate the checked-in demo. It must:
  1. `shutil.copytree` the demo into a temp dir.
  2. Mutate a tracked source so a region drifts — e.g. add a public method or
     change a documented signature in `engine.py` (a HASH/REGION drift on
     `core-api`).
  3. Run `cdx check` on the copy → show it reports drift (exit non-zero) and
     print the drift summary.
  4. Run `cdx monitor --apply` (mock backend, deterministic) → heal the region.
  5. Run `cdx check` again → clean (exit 0).
  6. Print the review log (the recorded verdict/provenance).
  7. Run `cdx rpt` → show the `scheduler.py` gap.
  8. Run `cdx doctor` → show the preflight PASS.
  - Drive cdx via the in-process CLI (typer) or subprocess; keep it OFFLINE and
    deterministic (inject `now`/use the mock backend; no network, no real git
    required — operate on the working tree / `mode=local` where git is needed).
  - Print clear section headers a human can follow. Exit 0 on success.
- Make it runnable: `python demo/walkthrough.py` from the repo root.
