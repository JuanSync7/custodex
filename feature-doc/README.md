# feature-doc/ — the golden feature reference (EPIC R)

This directory is the **single source of truth** for *every* feature of
code-doc-monitor. It exists so that demos, tests, and source can each be traced
back to a named, stable feature — and so a reader has one correct, complete
catalog of what the tool does.

## Layout

```
feature-doc/
├── catalog/            ← THE SOURCE OF TRUTH (machine-readable, hand-maintained)
│   ├── <subsystem>.yaml    one file per subsystem (extract, drift, heal, …)
│   └── …
├── FEATURES.md         ← RENDERED from catalog/ — do NOT hand-edit
├── wiki/               ← RENDERED from their single sources — do NOT hand-edit
│   ├── SOURCE_WIKI.md      per-module public symbols + feature links (srcindex.py)
│   ├── TEST_WIKI.md        every test case grouped by boundary (testwiki.py)
│   └── TRACEABILITY.md     the feature × demo × test matrix (traceability.py)
└── README.md           ← this file
```

- **`catalog/*.yaml`** is the source. Each file holds a `features:` list of
  `Feature` records (schema in `code_doc_monitor/featurecatalog.py`):
  `id` (`FEAT-<SUBSYSTEM>-<NNN>`, globally unique), `title`, `summary` (the golden
  prose), `subsystem`, `modules` (the real `code_doc_monitor` modules that
  implement it), `constraints` (K-refs upheld), optional `status`, and the
  optional `demos` / `tests` slots (currently unused — traceability is proven
  instead by inline `Feature: <id>` tags scanned from `tests/` and `demo/` by
  `cdmon trace`, not by populating these slots).
- **`FEATURES.md`** is rendered by `featurecatalog.render_features_md`, invoked
  via the `cdmon wiki` command (which also regenerates the `wiki/*` files). It is
  regenerated, never edited — the yaml is the only place a feature's prose lives,
  so there is nothing to drift.

## Why a typed catalog and not freeform Markdown

Prose duplicated between a yaml entry, a test docstring, and a rendered page would
drift exactly like code↔doc drift — the very thing cdmon exists to catch. So each
fact lives once and everything else is rendered. The catalog is loaded by a
validating loader (`featurecatalog.load_catalog`) that is **loud** (raises
`CatalogError`) on a duplicate id, a malformed entry, or a feature naming a
module that doesn't exist — the same discipline cdmon applies to the code it
monitors.

## Invariants (enforced by the loader / tests)

- Every feature `id` is unique and matches `FEAT-[A-Z][A-Z0-9]*-\d{3}`.
- Every `modules` entry is a real top-level module under `code_doc_monitor/`.
- The rendered `FEATURES.md` is a pure, deterministic function of `catalog/`.

## Current coverage

The catalog spans 19 subsystems (agent, backends, cli, config, configv2,
coverage, drift, extract, gitsync, heal, layout, learn, manifest, monitor, pr,
quality, record, reference, server). The exact feature count is the header of
`FEATURES.md` (199 at the time of writing), regenerated by `cdmon wiki` — see
`FEATURES.md` for the rendered view, so this prose never re-drifts on a catalog
addition.

## Regeneration & enforcement

Regenerate every artifact with `cdmon wiki` (renders `FEATURES.md` + the three
`wiki/*` files from their single sources). Two CI gates (the offline `docs:gate`
job) keep them honest:

- `cdmon wiki --check` — fails if any rendered wiki is stale vs its source.
- `cdmon trace --fail-on-gap` — fails if any catalogued feature lacks a demo or a
  test, by scanning inline `Feature: <id>` tags in `tests/` and `demo/` (currently
  199/199 complete).
